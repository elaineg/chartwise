# chartwise

Purpose: an explanation-first natal-chart web app for the curious computer worker — free, no signup, 100% client-side compute — that lays your chart out as a clean houses-as-rows table with a 1-2 sentence plain-English reading on every placement, plus a daily current-sky transit note AND a plain-English compatibility (synastry) reading between any two saved charts — the relationship feature other apps paywall, given away free.

Problem: the curious astrology-interested computer worker (marketers/designers/PMs skew here) who knows their Sun sign and wants to understand the rest without learning to read a wheel, plus friends-of-friends moments ("what's your big three?") that recur socially. Someone references rising signs / Mercury retrograde in Slack or at dinner; the person opens a tool to see what it means FOR THEM, cold, on phone or laptop. A person's own chart is computed once, so recurrence rests on two first-class MVP hooks: MULTI-PERSON charting (save profiles for partners, friends, dates, coworkers — done many times a month) and the DAILY CURRENT-SKY / TRANSIT reading (today's Moon/retrogrades vs your chart, the genuine return-visit habit).

Beats alternative: astro-seek.com (the primary free alternative) is comprehensive but DATA-DENSE — it assumes astrological literacy, presents positions/aspects as undefined reference data on a scattered wheel, and makes the user cross-reference separate "planet in house" pages (10-20 min per chart). co-star / the-pattern are polished but PAYWALLED and opaque (black-box, no chart transparency). chartwise wins by being EXPLANATION-FIRST and FREE: a houses-as-rows table with the plain-English meaning INLINED next to each placement, so the decoding work astro-seek pushes onto you is already done. That inline explanation layer is the differentiator and is core flow 2.

Core flows:
1. Create a profile and (optionally) save/share it. The user types birth date, birth time, and birth place in plain text; place resolves via a lazy-loaded bundled offline cities dataset (all-the-cities, type-ahead pick-from-suggestions; manual lat/long + optional UTC-offset escape hatch for unknown/historical cases). A named PROFILE is computed fully client-side with circular-natal-horoscope-js (Placidus houses, tropical signs, Sun-Pluto + Chiron, N/S nodes, retrograde flags, Asc/MC; the lib's Origin class derives IANA timezone + historical DST from lat/long internally). Profiles are saved to localStorage (multi-person list keyed on name, switch between them, no login). An optional "Create share link" writes the profile's birth inputs to Turso and returns a secret URL that, opened fresh, recomputes and renders the same chart (load-on-open, no polling). The SAME save/name/share affordance works for ESTIMATED (Big-3) charts (core flow 4) — one shared `name` field and one Turso share path serve both; a shared estimated chart reloads identically and stays labeled an estimate. (No "Load example (Einstein)" button — the natal landing example was removed from the UI 2026-06-26; the empty state prompts the user neutrally and Big-3 mode is the lowest-effort first chart.)
2. Read the chart as an explanation-first table. The chart renders as a STRUCTURED TABLE with the 12 HOUSES as rows (primary key) and Signs / Planets / Nodes as columns showing what falls in each house. An ELEMENT COUNT (Fire / Earth / Air / Water tally) shows as a compact bar. Every planet-in-house and sign-in-house cell, when opened, shows a CONCISE 1-2 sentence plain-English reading from the bundled in-house interpretation dataset (~310 original blurbs; a per-category "_generic" fallback covers any unfilled cell). A CURRENT-SKY / TRANSIT section below the chart computes today's planet positions client-side with the same lib, lists today's planet signs, flags retrogrades, and shows ≥1 concise note keying a notable current placement to the selected profile.
3. Compare two people — SYNASTRY / compatibility. The user picks TWO saved profiles (Person A, Person B) from an unmistakable two-person picker and gets a COMPATIBILITY view computed fully client-side from both charts, explanation-first and free (the feature Co-Star / The Pattern / astro-seek gate): (a) a full-width COMPATIBILITY SUMMARY — the big-three (Sun / Moon / Rising) of each person side by side, an element-balance comparison of the two charts, and a short HONEST harmony-vs-tension framing in plain English (NO fake "92% match" score); (b) a SYNASTRY INTER-ASPECT list — for each pair of bodies across the two charts, the major aspects (conjunction 0°, sextile 60°, square 90°, trine 120°, opposition 180°) within sensible orbs, each tagged harmony/tension and carrying a 1-2 sentence plain-English reading INLINE on the row; (c) a HOUSE-OVERLAY readout — which of A's houses each of B's planets falls into and vice-versa, each with a concise reading ("your Venus in my 7th house"). All readings come from the bundled in-house interpretation dataset (aspect + overlay blurbs, per-category "_generic" fallback).

4. Estimate a chart from just the BIG 3 (alternate entry, additive). Many people know their Sun / Moon / Rising signs from Co-Star but not their exact birth time or place. From the SAME input panel, a minimal secondary entry mode ("I only know my big 3") lets the user pick Sun sign, Moon sign, Rising sign (three dropdowns) plus a birth YEAR, and get an APPROXIMATE full chart (houses table, element counts, readings, transit note) WITHOUT the date/time/place form. Mechanism (fully client-side, reuses the existing `computeChart` ephemeris — no fork, no paid API): a deterministic SOLVER searches within the given year for a {date, time} at a FIXED reference location whose computed chart has Sun in the chosen sign, Moon in the chosen sign, and Ascendant in the chosen rising sign — Sun-sign bounds the ~30-day date window, Moon-sign narrows the day, Rising-sign picks the ~2-hour time slot. The inferred BirthData is then fed to the unchanged `computeChart`, so the rest of the chart (planets, houses, nodes, readings) populates exactly as the precise flow does. Because {big 3 + year} is UNDERDETERMINED, the result is explicitly labeled an ESTIMATE (date/time/place inferred, not yours) with a one-line note that entering full birth date/time/place gives the precise chart. An optional NAME field on the Big-3 form names the estimate (defaults to "Estimated chart" when blank); a named estimate is saved to localStorage and can be share-linked via the SAME `/api/chart-share` Turso path as precise charts (the `isEstimate` flag persists, so a reopened shared estimate re-renders the ESTIMATED CHART badge and is unmistakably an estimate). The precise birth-form entry (core flow 1) is unchanged; this is an additional option behind a small toggle.

Success checks:
- The page opens DIRECTLY into the birth form with NO hero headline, NO eyebrow micro-label ("Natal chart · Plain English · No signup"), and NO subtitle ("Free, no signup — type your birth date, time, and place."). There is no awkward gap at the top — the entry-mode toggle (top of the birth form) appears at normal top-of-content spacing. The three hero copy elements are entirely absent from the rendered HTML body.
- No "Load example (Einstein)" button or any user-facing "Einstein" example appears ANYWHERE in the UI (landing, empty state, hero/subtitle, placeholders, help text). Internal test/regression anchors in `*.test.ts`/`*.spec.ts` may reference Einstein, but no product surface renders it. The PRECISE mode leads directly with the birth form (no example button, no orphaned "OR / Or enter your own" divider); the empty state gives a neutral prompt telling the user what to do.
- Estimate a Big-3 chart (Sun = Aries, Moon = Taurus, Rising = Gemini, year = 1990): the houses table renders, the element-count bar shows numeric Fire / Earth / Air / Water totals that sum to the number of placements tallied (visible four-way breakdown, not blank), and the element-distribution bar displays a "Based on N placements: Sun, Moon, Mercury…" basis label naming the counted bodies.
- Opening any planet-in-house cell on a rendered chart shows a 1-2 sentence plain-English reading; opening a sign-in-house entry likewise shows a 1-2 sentence reading.
- The Current-sky / transit section is present and lists today's planet signs, flags any retrogrades, and shows at least one 1-2 sentence note tied to the loaded profile (structure/presence check — values are today-dependent and not asserted exactly).
- Entering a custom birth date/time and a place via the type-ahead (e.g. start typing a city, pick a suggestion) produces a named profile and a populated houses table without any network call to a chart/geocoding API.
- Creating a share link yields a URL that, opened in a fresh session with no localStorage, renders the same houses-as-rows table for that profile.
- BIG-3 FORM SPACING: in the BIG 3 tab, the explainer, the three sign dropdowns, the birth-year input, and the Estimate button each sit in a consistent vertical rhythm (equal row→row spacing, equal label→control gap) with generous whitespace — no two adjacent field groups have visibly different gaps, and nothing is cramped (uses the shared `.ds-field` spacing, --sp-6 between groups).
- CONSISTENT BOX PADDING: on the estimated-chart result view, EVERY bordered box/card (profile summary, the ESTIMATED CHART badge, the "Compare two people / Compatibility" entry, the Share card) has equal interior padding on ALL FOUR sides — no text touches any border (in particular the "Compare two people" box text is NOT flush against the left border). Verifiable at desktop and 375px; no `var(--sp-5)` (an undefined → 0 token) remains in the styles.
- SAVE + NAME ESTIMATED CHART: estimate a Big-3 chart, give it a name in the Big-3 form's NAME field, and create a share link → the URL, opened in a fresh session with no localStorage, reloads the SAME Big-3 result (same Sun/Moon/Rising signs and houses table) under the entered name AND shows the ESTIMATED CHART badge, clearly distinguishing it from a precise chart. With the name left blank, estimating still works and the chart saves as "Estimated chart" (naming never gates the Estimate action). The estimated save reuses the existing `/api/chart-share` Turso path and the single `BirthData.name` field — no second/parallel name field exists.
- The app loads and computes a chart with the network/DB offline (chart compute is fully client-side; only share-link creation/open touches Turso).
- Lunar nodes (North Node / South Node) display their own sign and degree (e.g. "North Node 2°07' Virgo"), not the sign of the house they occupy. For the Jiangmen/1998-08-08 QA anchor chart, the axis reads Virgo–Pisces (not Leo–Aquarius).
- Black Moon Lilith (Mean) is displayed as a body placement. For the Jiangmen QA anchor chart it appears at ~26° Libra, House 10.
- All body positions are shown with arcminutes (degree + minutes format, e.g. "23°52' Cancer"), not floor-truncated whole degrees.
- No "Save as image" button is present in the UI.
- Visual language reads as austere SSENSE editorial minimalism — monochrome (no element/zodiac/indigo/violet/amber colors, no shadows, no gradients), square corners, 1px hairline rules, tiny uppercase letter-spaced micro-labels, and generous whitespace — while all chart data, the houses-as-rows table, every reading, the element tally, accuracy (nodes/Lilith/arcminutes), and all flows are unchanged (per DESIGN_REDESIGN.md / lib/design-system/ssense.md).

Synastry / compatibility (core flow 3):
- A first-class "Compare two people" entry is visible from the chart view WITHOUT scrolling past the natal chart (a primary, labeled action — not buried below the transit card), with a clear sub-label naming the value (e.g. "Plain-English compatibility between two charts").
- The compatibility view has an unmistakable TWO-PERSON picker: Person A and Person B are each explicitly chosen from the saved-charts list, both selections are visible at once, and the view recomputes when either changes. With fewer than two saved charts, the picker shows a "Load example pair" affordance so the payoff is reachable in one click.
- A one-click "Load example pair" seeds two known charts (Einstein + a fixed second dataset, both time-invariant) and renders the full compatibility view — the worked example a cold visitor sees before saving anyone.
- REGRESSION ANCHOR (deterministic, stranger-verifiable in the browser): load the example pair — Person A = Albert Einstein (14 March 1879, 11:30 LMT, Ulm) and Person B = Michelle Obama (17 January 1964, 21:53 CST, Chicago — AstroDatabank A-rated birth data). The SYNASTRY INTER-ASPECT list shows a Moon–Jupiter TRINE between Einstein's Moon (Sagittarius, ~254.3°) and Michelle's Jupiter (Aries, ~12.7°), with an orb of ~1.67° (≤2°), tagged HARMONY, with a 1-2 sentence plain-English reading on the row. The house-overlay section shows Einstein's planets placed into Michelle's Placidus houses (and vice versa) using the SAME Placidus cusp degrees as the natal chart — for any self-synastry the overlay house must equal the natal house exactly.
- The COMPATIBILITY SUMMARY shows BOTH people's big three side by side: for the example pair, Person A (Einstein) reads Sun Pisces / Moon Sagittarius / Rising Cancer, and Person B (Michelle Obama) reads Sun Capricorn — both columns visible at once, un-clipped, without a click.
- Every inter-aspect ROW and every house-overlay ROW is visible WITHOUT a click (no expand-to-reveal gate on the readings) and is un-clipped/un-truncated at BOTH desktop and 375px — long aspect/overlay labels WRAP, never truncate to an ellipsis.
- Harmony vs tension is distinguishable AT A GLANCE in monochrome (no color): each aspect row carries an explicit text tag (e.g. HARMONY / TENSION or trine/square wording) — never color alone — and aspects are grouped/ordered so the reader sees the harmony-vs-tension balance without reading every row.
- The element-balance comparison between the two charts stays glanceable in monochrome via dominant-first ORDER + grey-VALUE ramp + LENGTH = proportion (matching the existing ElementBar convention), never identical mono fills.
- The compatibility view reads as SSENSE: uppercase 11px tracked micro-labels, 1px hairline rules, square corners, no shadows/color/gradients, the existing `.ds-*` classes; it computes with the network offline (no chart/geocoding API call).

Big-3 estimate (core flow 4):
- A minimal secondary entry mode is reachable from the input panel WITHOUT replacing or altering the precise birth-form: a small, discoverable toggle/tab labeled to the effect of "I only know my big 3" switches the panel to three sign dropdowns (Sun / Moon / Rising) + a birth-year field + an "Estimate chart" action. The precise form remains the default and is one click away.
- CORRECTNESS INVARIANT (deterministic, stranger-verifiable): pick Sun = Aries, Moon = Taurus, Rising = Gemini, year = 1990, Estimate → a chart renders whose Sun sign reads Aries, Moon sign reads Taurus, and Ascendant sign reads Gemini. The estimated chart MUST honor all three entered signs (a chart that silently disagrees with the user's own inputs is a P0). The solver hits all three for every valid combination (all 12×12×12 are reachable at the reference latitude).
- The estimated chart is CLEARLY labeled an ESTIMATE/approximation (e.g. an "Estimated chart" badge or banner on the result), with a one-line note that its date, time, and place were inferred and that entering full birth date/time/place gives the precise chart.
- The estimate computes with the network/DB offline (same client-side ephemeris; no chart/geocoding API call) and adds no new dependency.
- Switching to the big-3 mode and back leaves the precise birth-form flow unchanged: the PRECISE tab still shows the birth-info form (date/time/place + type-ahead) and computing a precise chart from it renders the same houses-as-rows table as before (core-flow-1 regression unaffected). The mode toggle swaps only the form body; an already-rendered chart on the right is not cleared by toggling.
- An optional NAME field appears in the BIG 3 form; estimating with a name set saves the estimate under that name in the people switcher, and estimating with it blank saves as "Estimated chart" — the Estimate action succeeds in both cases.
- The big-3 entry UI reads as SSENSE: uppercase tracked micro-labels, 1px hairline rules, square corners, existing `.ds-*` classes, no color/shadow/gradient; the toggle and dropdowns match the existing form controls.

Out of scope: the natal "Load example (Einstein)" landing button (REMOVED from the UI 2026-06-26 — Einstein survives only as internal test/regression anchors, never as a product surface; the synastry "Load example pair" compatibility seed is a separate affordance and unaffected by this removal); the circular chart wheel drawing (we lead with the table — a wheel is at most a later nice-to-have); a numeric compatibility "percentage/score" gimmick (we frame harmony vs tension in honest plain English, never a fake number); composite/Davison midpoint charts (synastry is cross-aspects + overlays only for MVP); accounts/login/auth; any paid feature or paid API; push notifications or email; modalities beyond the basic element count if time-constrained (elements are required, modalities optional); save-as-image / chart image download (removed — the canvas approach produced no useful artifact).
